# Figma Design-System File and Component Handoff Standard v0.6

Status: Draft local standard

Use this standard to make the whole Figma design-system file governable and
each component area clear for design iteration, developer handoff, and
AI-assisted implementation. It is a local operating standard; public design
systems do not share one universal Figma file hierarchy.

## How to Use This Standard

1. Organize the file using the file-wide model below.
2. Give every production component or closely related family its own top-level
   Component Section.
3. Paste the prompt at the end into Figma's built-in agent and answer its setup
   questions.
4. Review the audit before changing canonical sources.
5. Use the remediation prompt generated by the audit to create a reviewable
   draft of one scoped improvement.

The audit covers the entire file. Remediation remains deliberately scoped so a
single change cannot silently reorganize unrelated library work.

## The Operating Hierarchy

```text
Figma file              governed design system or library scope
├── Page                 navigation and grouping
│   ├── Section          review, iteration, and handoff contract
│   │   ├── Frame        structured part of that contract
│   │   ├── Component    canonical owned source
│   │   ├── Instance     reference, specification, or example
│   │   └── Annotation   durable non-visual guidance
│   └── Section          another component, foundation, or pattern
└── Variables/styles     shared foundations used by canonical sources
```

- The **file** owns system direction, source mapping, governance, lifecycle,
  and evidence of system health.
- A **Page** organizes related Sections. It is not normally the component
  contract or handoff unit.
- A top-level **Section** is the formal node for what the team may informally
  call an artboard. It is independently linkable and is the normal review and
  handoff unit.
- A Page may contain many Sections. Give unrelated work separate Sections even
  when it remains on the same Page.
- A canonical **Component** or **Component Set** is the design source of truth
  for an owned component. The surrounding Section explains its contract.

## Recommended File Shape

The logical areas are more important than exact Page names or count. Large
libraries may split one area across several Pages or linked files.

```text
File: <Design system>
├── Page: 00 / Start Here
│   ├── Section: System / Direction & Scope
│   ├── Section: System / Architecture & Sources
│   ├── Section: System / Ownership & Contribution
│   ├── Section: System / Lifecycle & Change
│   └── Section: System / Adoption & Health
├── Page(s): Foundations / <Area>
│   └── Section: Foundation / <Name>
├── Page(s): Components / <Group>
│   ├── Section: Component / <Name>
│   └── Section: Component / <Related family>
├── Page(s): Patterns / <Domain> — when applicable
│   └── Section: Pattern / <Name>
├── Page(s): Workbench / <Area>
└── Page(s): Archive / <Area>
```

Each Page needs a clear primary purpose. Component Pages may contain as many
component Sections as remain navigable. Do not force one component per Page.

If governance, code, release, support, or research sources live outside Figma,
link the authoritative source rather than copying it into the file. Missing
external access must be recorded as unknown, not guessed.

## File-Wide Governance Contract

The Start Here area must make the following discoverable in Figma or through
direct authoritative links.

### Direction & Scope

- system purpose, audiences, supported products and platforms;
- principles and material constraints;
- what the system owns and explicitly does not own;
- relationship to other design systems, brands, or product libraries.

### Architecture & Sources

- Page and library map;
- canonical Components, Component Sets, variables, styles, and assets;
- external design dependencies;
- code packages, source paths, documentation, and Code Connect mappings;
- which source is authoritative when design, code, and documentation differ.

### Ownership & Contribution

- accountable system owner and component owners where known;
- how work is proposed, reviewed, approved, and published;
- required design, engineering, product, accessibility, and brand review;
- decision and escalation path for unresolved questions.

### Lifecycle & Change

Keep these three axes separate:

- lifecycle: Planned, Experimental, Active, Deprecated, or Archived;
- design handoff: Draft, Ready for dev, or Changed;
- implementation mapping: Complete, Partial, Missing, or Unknown.

Record meaningful releases and changes. A deprecated source must identify its
reason, affected dependencies, approved replacement, migration guidance,
support window when known, and removal criteria.

### Adoption & Health

- support and feedback channels;
- known systemic gaps, defects, and design-code drift;
- available adoption, accessibility, quality, or maintenance evidence;
- current improvement priorities and their decision owners.

Do not invent metrics merely to fill this area. Link real evidence and mark
missing inputs explicitly.

## Production, Workbench, and Archive Boundaries

- **Production** Sections contain canonical sources and current contracts.
- **Workbench** Sections contain exploration that is not approved for handoff.
- **Archive** Sections contain superseded material and point to its replacement
  when one exists.
- Detached Instances, copied masters, and unlabeled explorations must not look
  authoritative.
- Moving work between these areas requires an explicit lifecycle decision.

## Foundation Sections

Foundations include variables, styles, assets, and rules such as color,
typography, spacing, elevation, motion, grids, icons, and themes.

```text
Section: Foundation / <Name>
├── 01 Overview
├── 02 Canonical Sources
├── 03 Rules & Modes
└── 04 Examples
```

Variables and styles are the source of truth for values. The Foundation Section
explains purpose, semantic relationships, modes, constraints, and platform
exceptions. Do not create a second manual token catalogue that can drift from
the real variables.

## Component Section Contract

The handoff unit is one top-level Section for a component or a closely related
component family. Use a family Section only when its components share a
contract or must be understood together, such as Input and Field.

```text
Section: Component / <Name>
├── 01 Overview
├── 02 Components
│   ├── Public
│   │   ├── Primitive / <Name>
│   │   └── Compound / <Name>
│   └── Internal — when applicable
├── 03 Specification
│   ├── Anatomy & Composition
│   ├── Properties & Defaults
│   ├── Variants, Sizes & State Model
│   ├── Behavior & Interaction
│   ├── Accessibility & Semantics
│   └── Content, Layout & Localization — when applicable
└── 04 Examples
```

The four numbered items are direct child Frames. Include only classifications
and Specification subsections that apply.

### 01 Overview

Keep this short:

- purpose and when to use or not use;
- public components owned here and external dependencies;
- target platforms when contracts differ;
- owner, lifecycle, handoff status, known limitations, and open decisions.

### 02 Components

Every public component owned by the Section belongs under `Public`:

- `Primitive / <Name>` is independently usable and has one focused
  responsibility in this system, such as Input, Button, Textarea, or Option.
- `Compound / <Name>` is also a public component, but is built from nested
  canonical components and owns a stable relationship or behavior, such as
  Field, InputGroup, OTPGroup, or ButtonGroup.

`Internal` contains deliberately reusable construction components that are not
public API. Ordinary layers remain named anatomy; do not create Internal
Components merely to fill the group.

Keep one canonical source for each owned component. List used-but-not-owned
components as dependencies. Use nested Instances, preserve variable bindings,
and name layers by responsibility, such as `control`, `label`, `description`,
`error`, `leading`, `trailing`, `trigger`, `value`, and `action`.

A canonical source may live in this Section or elsewhere. When it lives
elsewhere, show a linked Instance and provide an exact main-component link.
Specification coverage and Examples use linked Instances, never detached copies
or duplicated masters. Move durable decisions out of comments and into
properties, descriptions, annotations, or the appropriate Specification Frame.

For a Compound, identify:

- wrapper and nested canonical components;
- optional, replaceable, repeatable, or freeform regions;
- allowed content and empty behavior;
- ownership of layout, state, behavior, and accessibility.

Use instance swap with preferred values when one item is chosen from known
compatible components. Use a Figma Slot only for intentional freeform content;
document allowed content, quantity, order, empty behavior, and layout ownership.

#### Field example

```text
02 Components
├── Public
│   ├── Primitive / Input
│   └── Compound / Field
│       ├── label          text property
│       ├── control        nested canonical Instance
│       ├── description    optional text property
│       └── error          optional text property
└── Internal — only when reusable Internal Components exist
```

Field owns the label, description, error, required indication, layout around
one control, and their semantic relationships. The nested control owns its
value, placeholder, icons, native behavior, and control-specific states.

The contract must state that the label identifies the control, description and
error describe it, invalid state is exposed programmatically, and exactly one
compatible control is accepted. Related controls become a named Compound such
as InputGroup, or FieldGroup when `fieldset` and `legend` semantics are owned.

### 03 Specification

Use linked Instances, concise notes, annotations, and prototypes where each
communicates best.

- **Anatomy & Composition:** meaningful parts, nested components, flexible
  regions, and layout ownership. Do not promote ordinary anatomy into
  components.
- **Properties & Defaults:** the canonical Component or Component Set is the
  source of truth. Add notes only for meaning, constraints, or code mapping
  that properties cannot express; do not maintain a duplicate API table.
- **Variants, Sizes & State Model:** show every supported value at least once
  and important combinations. Separate variant, size, control, validation,
  interaction, selection, and disclosure axes when they can combine. Document
  precedence and unsupported combinations; avoid a full Cartesian matrix
  unless review genuinely requires it.
- **Behavior & Interaction:** activation, dismissal, keyboard and focus,
  transitions, loading, validation, selection, overflow, responsiveness, and
  meaningful motion when applicable.
- **Accessibility & Semantics:** semantic element or role, accessible name and
  description, keyboard behavior, focus, element relationships, announced
  states, contrast, zoom, high contrast, reduced motion, and target size when
  applicable. Never infer semantics from appearance.
- **Content, Layout & Localization:** writing rules, wrapping or truncation,
  width constraints, responsive behavior, long translations, RTL, locale
  formatting, themes, and variable modes when applicable.

Canonical layers use existing variables. Annotate only raw values, missing
variables, non-obvious modes, or Figma-variable-to-code-token differences.

### 04 Examples

Build Examples from linked canonical Instances. Add as many as useful:

- basic use and component compositions;
- realistic product contexts and states in context;
- long content, narrow and responsive layouts;
- localization, RTL, themes, and variable modes;
- do and do-not comparisons.

Promote an Example to a public Compound only when it has a stable reusable
contract, configuration, behavior or accessibility ownership, and repeated use.

## Pattern Sections

A Pattern teaches a repeatable product arrangement without claiming a public
library component API.

```text
Section: Pattern / <Name>
├── 01 Overview
├── 02 Composition
├── 03 Behavior & Accessibility
└── 04 Examples
```

Patterns use linked component Instances. When a composition gains an approved
canonical source, stable API, and owned behavior or accessibility contract,
promote it to a public Compound component.

## Figma-to-Code Boundary

Figma defines the visual and framework-neutral design contract, not the exact
API for every framework. Reliable implementation also needs the target
platform, package conventions, source path, property mapping, imports, usage
examples, and behavior and accessibility tests.

Use Code Connect when available. Do not claim unambiguous code generation when
implementation mapping is missing, even if the design contract is complete.

## Readiness Model

Assess every production Component Section on two independent outcomes:

- **Design contract:** Ready or Not ready.
- **Implementation mapping:** Complete, Partial, Missing, or Unknown.

`Ready with documented implementation gaps` is a useful combined summary when
the design contract is Ready but implementation mapping is not Complete.

A design contract is Ready only when canonical sources, ownership, anatomy,
properties, defaults, supported states, composition, behavior, accessibility,
variables, and examples are sufficiently explicit and internally consistent.
Visual completeness alone is not readiness.

File governance health and component readiness inform one another but do not
replace one another. A component may be design-ready while the wider system
lacks adoption evidence, and a well-governed file may still contain incomplete
component contracts.

## Iteration and Remediation

Designers iterate using canonical sources and Workbench Sections. Before an
agent performs structural remediation, create a sibling draft Section named:

`Component / <Name> — AI-ready draft`

Do not duplicate the whole Page when only one Section is in scope. Preserve
the original Section, canonical links, nested Instances, and variable bindings.
Record what changed, what remains unresolved, and which changes require
approval. Merge approved work through the normal contribution workflow.

## Transitioning Existing Component Areas

Reorganize correct work rather than rebuilding it:

| Existing content | Destination |
| --- | --- |
| Purpose, ownership, status, limitations | `01 Overview` |
| Owned independently usable source | `02 Components → Public → Primitive` |
| Owned source composed from components | `02 Components → Public → Compound` |
| Private reusable construction component | `02 Components → Internal` |
| Anatomy and composition | `03 Specification → Anatomy & Composition` |
| Variant, size, and state demonstrations | `03 Specification → Variants, Sizes & State Model` |
| Keyboard, focus, behavior, semantics | Relevant `03 Specification` subsection |
| Product arrangements, themes, stress cases | `04 Examples` |
| Token usage | Canonical variable bindings; exceptions in `03 Specification` |

For Button, resolve action versus navigation semantics; tone, variant, size,
icon-only, and loading decisions; grouping; focus; disabled behavior; naming;
and overflow without turning visual examples into accidental APIs.

For Input and related controls, keep independently supported controls as public
Primitives; make Field and stable input groups public Compounds; keep ordinary
construction layers as anatomy; state Field-to-control semantics explicitly;
separate control, validation, and interaction state axes; and leave currency,
phone, country, or other arrangements in Examples until they have an approved
reusable contract.

## Handoff Checklist

- [ ] The file exposes direction, sources, ownership, lifecycle, and support.
- [ ] Every Page and production Section has a clear purpose and classification.
- [ ] The URL selects the exact top-level component Section.
- [ ] The four numbered component Frames exist.
- [ ] Owned sources, external dependencies, and examples are distinguishable.
- [ ] Public Primitive, Public Compound, and Internal boundaries are clear.
- [ ] Compounds use nested canonical Instances and explicit ownership rules.
- [ ] Properties, defaults, state axes, combinations, and precedence are clear.
- [ ] Behavior, keyboard, focus, semantics, and accessibility are documented.
- [ ] Canonical layers use variables and disclose token gaps.
- [ ] Target platforms and Figma-to-code mapping status are recorded.
- [ ] Lifecycle, design handoff, and implementation status are not conflated.
- [ ] No material design or product decision is hidden or silently guessed.

## Basis and Confidence

The content model distills recurring practices from public government,
enterprise, open-source, and premium-brand design systems. The canvas mechanics
follow Figma's native Pages, Sections, component properties, nested Instances,
variables, annotations, Dev Mode status, and Code Connect. Confidence is high
in this local operating model, not in a claim that one universal industry
canvas standard already exists.

## Copy-Paste Whole-File Audit Prompt for Figma's Built-In Agent

Paste the following self-contained prompt into Figma's built-in agent:

```text
Act as a senior design-system auditor inside the currently open Figma file.
Audit the whole design-system file and every production component Section. The
audit must cover file-wide governance and component handoff readiness together.

Do not edit, move, rename, detach, recreate, publish, delete, or redesign any
existing source node during the audit. The only permitted file mutation is
creating or appending to the audit-report Page described below. If that is not
possible, return the same report in chat and state that no audit Page was made.

Before inspecting or writing, respond only with:

Please provide:
1. Design-system name and whether the whole current file is in scope. If not,
   list excluded Pages.
2. Supported products and target implementation platforms.
3. Direct links to known code packages, documentation, repositories, or other
   authoritative sources. Say "unknown" when unavailable.
4. Whether Code Connect or another design-to-code mapping exists. Say
   "unknown" when unavailable.
5. Known constraints, active migrations, or decisions that must be preserved.
   Say "none" when there are none.
6. Permission to create or append to `00 — AI-ready design-system audit`.

Wait for the answers, then inspect the entire in-scope file. Never invent
missing owners, metrics, decisions, code mappings, or external evidence. Use
`Unknown` or `Unassigned` and explain what is needed.

OPERATING MODEL

- The Figma file is the governed system or library scope.
- Pages provide navigation and grouping. A Page may contain many component,
  foundation, pattern, workbench, or archive Sections.
- A top-level Section is the review, iteration, and handoff unit.
- Frames structure a Section.
- Components and Component Sets are canonical owned design sources.
- Instances are references, specification coverage, and examples.
- Variables and styles are canonical foundation values.

Do not assume one component per Page. Inventory Pages, but assess production
readiness at the Component Section level. Group Section findings under their
source Page.

RECOMMENDED FILE AREAS

- Start Here: direction and scope; architecture and sources; ownership and
  contribution; lifecycle and change; adoption and health.
- Foundations: canonical variables, styles, assets, rules, and modes.
- Components: one top-level Section per component or closely related family.
- Patterns: repeatable arrangements that are not public component APIs.
- Workbench: unapproved exploration.
- Archive: superseded material with replacement links when applicable.

Exact Page names and Page count may differ. Assess whether these responsibilities
are discoverable in Figma or through direct authoritative links; do not require
duplicated documentation.

FILE-WIDE AUDIT

Assess:
1. Direction, audiences, platforms, principles, scope, and system boundaries.
2. File architecture, Page purposes, library map, and canonical source map.
3. Foundation integrity: variables, styles, assets, modes, raw values, and
   semantic versus primitive usage.
4. Component and pattern ownership, naming, duplication, and dependency
   boundaries.
5. Design, documentation, code, Code Connect, and test alignment.
6. Ownership, contribution, review, approval, and decision paths.
7. Lifecycle, release, change, deprecation, migration, and archive practices.
8. Support, feedback, adoption, accessibility, quality, maintenance, and known
   system-health evidence.

Keep lifecycle, design handoff, and implementation mapping separate:
- Lifecycle: Planned / Experimental / Active / Deprecated / Archived / Unknown.
- Design handoff: Draft / Ready for dev / Changed / Unknown.
- Implementation mapping: Complete / Partial / Missing / Unknown.

COMPONENT SECTION STANDARD

Use this target hierarchy for every production component or related family:

Section: Component / <Name>
├── 01 Overview
├── 02 Components
│   ├── Public
│   │   ├── Primitive / <Name>
│   │   └── Compound / <Name>
│   └── Internal — when applicable
├── 03 Specification
│   ├── Anatomy & Composition
│   ├── Properties & Defaults
│   ├── Variants, Sizes & State Model
│   ├── Behavior & Interaction
│   ├── Accessibility & Semantics
│   └── Content, Layout & Localization — when applicable
└── 04 Examples

The four numbered items are direct child Frames. Include only applicable
classifications and Specification subsections.

For each Component Section check:
- purpose, use and non-use, owner, lifecycle, platforms, dependencies, limits,
  and unresolved decisions;
- exactly one canonical source for every owned component;
- owned Public Primitive, owned Public Compound, Internal, and external
  dependency classifications;
- no detached copies or duplicate masters presented as sources;
- anatomy, nested canonical Instances, flexible regions, allowed content, and
  layout, state, behavior, and accessibility ownership;
- clear component properties, values, defaults, and unsupported combinations;
- separate variant, size, control, validation, interaction, selection, and
  disclosure axes when they can combine; do not require a full Cartesian
  matrix;
- applicable activation, dismissal, keyboard, focus, transitions, loading,
  validation, overflow, responsive, and motion behavior;
- applicable semantic element or role, accessible naming and description,
  element relationships, announced states, contrast, zoom, high contrast,
  reduced motion, and target size;
- applicable content rules, long translations, RTL, themes, modes, width, and
  responsive behavior;
- canonical variable bindings and disclosed raw values, missing variables,
  mode ambiguity, or design-to-code token differences;
- linked canonical Instances for specification and examples;
- target package, source, imports, property mapping, Code Connect, usage, and
  behavior and accessibility tests when implementation evidence is available;
- change, deprecation, replacement, and migration information when applicable.

A Primitive is an independently usable public component with one focused
responsibility. A Compound is an equally public component made from nested
canonical components and owning a stable relationship or behavior. Internal
contains only deliberately reusable non-public construction components;
ordinary layers remain named anatomy.

For Field specifically, verify that Field is a public Compound owning label,
description, error, required indication, layout around one control, and their
semantic relationships. Its nested control owns value, placeholder, icons,
native behavior, and control-specific states. The contract must say that the
label identifies the control, description and error describe it, invalid state
is exposed programmatically, and exactly one compatible control is accepted.

Use instance swap with preferred values for one known compatible nested item.
Use a Slot only for intentional freeform content and require allowed content,
quantity, order, empty behavior, and layout ownership.

READINESS

Report two independent outcomes for every production Component Section:

Design contract: Ready / Not ready
Implementation mapping: Complete / Partial / Missing / Unknown

Use the combined summary `Ready with documented implementation gaps` when the
design contract is Ready and mapping is not Complete. Do not mark a design
contract Ready when material purpose, ownership, canonical-source, anatomy,
property, state, behavior, accessibility, composition, variable, or product
decisions remain unclear or conflicting. Visual completeness alone is not
readiness. File-governance gaps do not automatically make every component Not
ready; report the direct impact and evidence.

AUDIT REPORT

If permitted, use Page `00 — AI-ready design-system audit`. If it already
exists, append a new dated Section; do not overwrite earlier audit runs.

Section: Audit / <file name> / <date>
├── 01 Executive Summary
├── 02 File Governance
├── 03 Page & Section Inventory
├── 04 Component Handoff Findings
├── 05 Cross-System Findings
├── 06 Prioritized Actions
└── 07 Open Decisions & Missing Context

Inventory every in-scope Page and its top-level Sections. Record exact names,
node links, primary purpose, classification, lifecycle when evidenced, and
whether canonical sources appear present. A Page may have multiple compatible
classifications because its Sections are the actual contracts.

Create detailed findings for every production Component Section, grouped by
source Page. For each include:
- exact Section name and node link;
- owned sources and external dependencies;
- lifecycle, design-handoff, design-contract, and implementation-mapping
  results with one-sentence evidence-based reasons;
- what is already clear;
- material gaps with exact evidence links;
- the recommended Section hierarchy using only applicable subsections;
- actionable remediation, acceptance criteria, dependencies, and decision
  owner or Unassigned;
- what can be implemented without guessing and what remains unsafe to assume.

For Foundation, Pattern, governance, Workbench, Archive, utility, or unclear
areas, provide the inventory entry and only material detailed findings. Do not
create empty repetitive audit Frames merely to prove coverage.

Consolidate cross-system duplication, competing sources, unclear ownership,
inconsistent naming or properties, variant multiplication, detached assets,
token misuse, accessibility gaps, stale content, lifecycle drift, and missing
design-code mappings. Do not claim a root cause without evidence.

Prioritize actions as:
- P0: blocks correct, safe, accessible, or authoritative use.
- P1: blocks reliable reuse, composition, parity, or handoff.
- P2: improves clarity, consistency, adoption, or maintenance.

Every action must state target, reason, expected result, acceptance criteria,
dependencies, and owner or Unassigned. Do not invent effort, deadline, owner,
or final priority authority.

If limits prevent complete inspection, visibly mark `Audit incomplete`, list
everything inspected, identify the first unaudited Page or Section, and retain
the report structure so work can resume. Never claim complete coverage without
evidence.

RETURN IN CHAT

Return:
1. Pages and top-level Sections inspected and any not inspected.
2. Count of production Component Sections.
3. Design Ready and Not ready counts.
4. Implementation Complete, Partial, Missing, and Unknown counts.
5. The five most important file-wide or cross-system findings.
6. The recommended first three to five actions.
7. A direct link to the audit Section.
8. One self-contained `NEXT REMEDIATION PROMPT` for the recommended first
   scoped target.

The remediation prompt must include the exact target node link, audit evidence,
preserved constraints, target hierarchy, required acceptance criteria, and
unresolved decisions. It must instruct the agent to create a sibling draft
Section named `<target Section name> — AI-ready draft`, preserve the original
source and variable bindings, avoid unrelated redesign, record a change log,
and stop for approval before altering canonical sources. If the first priority
is file governance or a Foundation rather than a Component, generate an
equivalently scoped draft-Section prompt using the correct target type.

Stop after the audit and generated remediation prompt. Do not remediate source
work until that prompt is separately approved and run.
```
