**Token And Geometry Guardrails**

- Reuse existing semantic/component variables from Prompt 01 wherever they exist.
- Use slash-separated Figma variable paths such as `component/input/icon/size`, not source names like `component/input/icon-size`.
- Component color tokens should stay semantic. Do not bind component colors directly to primitive palette variables.
- Component geometry tokens may alias primitive size/spacing variables through component paths. For example, `component/input/icon/size` may alias `space/05`.
- Do not replace a bound variable with a raw value while removing labels/descriptions/errors.
- If a needed variable is missing, report it instead of inventing a raw value or silently creating a different naming convention.
- Approved width tokens may exist, but primitive controls should not force layout width internally unless width is intrinsic to the component behavior or visual contract. Width tokens are usually consumed by Field, Field Group, Storybook examples, forms, or application layouts.
