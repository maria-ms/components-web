# Verify Composability

```text
You are working in the Design System v2.0 2026 Figma file.

Goal:
Run a final composability audit after the input page has been classified and reorganized.

Do not make broad visual changes in this pass. Inspect first, then report.

Context:
The design system should be portable into code by exposing primitives and structural wrappers. Product-specific use cases should be examples unless they have stable reusable behavior.

The expected code mapping is:

- Primitive input controls map to Web Components such as ds-input-select and ds-input-number.
- Field maps to ds-field.
- Compound grouped controls map to a reusable wrapper only when shared chrome/focus grouping is needed.
- Currency, phone, country, and card examples usually stay as patterns composed by consumers.

Audit these principles:

1. Single responsibility

Each primitive should do one job:
- text input enters text
- select input chooses from options
- number input enters/steps numbers
- OTP input handles OTP-specific segmented entry

Fail anything that mixes product/domain behavior into a primitive.

2. Labels and descriptions

Primitive controls should not contain:
- visible label
- visible description/help text
- visible error message

These belong to field or field examples.

3. Composition

Patterns should demonstrate composition instead of creating rigid components.

Currency should be explainable as:
- field + compound control + select + number/text input

Phone should be explainable as:
- field + compound control + select + text input

Country should be explainable as:
- field + select

Card/payment input should be explainable as:
- field + text input + consumer-provided adornment/media

4. State model

Primitive controls should retain their own native-like states:
- default
- hover, if designed
- focus/focus-visible
- disabled
- invalid/destructive
- filled

Field should handle field-level context:
- label state
- description state
- error state
- invalid/error display

Compound examples may show child focus states, but those should be documented as child focus behavior, not as product-specific public component variants.

5. Token readiness

Confirm that primitives and wrappers are bound to variables for:
- color
- typography
- spacing
- radius
- border
- shadow
- height
- icon size
- component-specific internal geometry

If a value exists only as raw geometry in Figma and should be consumed by code, flag it as needing a component variable.

6. Naming

Confirm names communicate classification:
- primitives use neutral component names
- wrappers use structural names
- examples use Pattern, Playground, or Example in their section or label
- product/domain examples are not named like exported package components

Report format:

- Overall PASS / NEEDS REVIEW / FAIL
- Classification problems remaining
- Composition problems remaining
- Token-binding problems remaining
- Naming problems remaining
- Clear final recommendation: ready for code pipeline or needs another Figma cleanup pass
```
