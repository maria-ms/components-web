# Restructure Input Page

```text
You are working in the Design System v2.0 2026 Figma file.

Goal:
Reorganize the input components page so designers and engineers can clearly distinguish primitives, structural wrappers, internal parts, and product patterns.

Context:
The code design system should expose the smallest reusable building blocks needed to build product interfaces consistently.

The desired page model is:

1. Primitives
Reusable components with one clear responsibility and durable behavior.

2. Structural wrappers
Reusable composition shells that provide field layout, label/description/error placement, accessibility wiring, or grouped-control visual behavior.

3. Internal parts
Atoms or subcomponents used to build public primitives/wrappers.

4. Patterns / playgrounds
Examples showing how primitives and wrappers compose into product-like use cases.

Before making changes:
- Use the result of the classification audit.
- Do not change visual styles unless a style is clearly attached to the wrong component classification.
- Do not delete design evidence.
- Preserve useful examples, but move them into the right section.

Please reorganize the input page into clear sections:

## Primitives

Place primitive controls here:
- input/default or base input shell
- input/text
- input/select
- input/number
- input-otp, if it is intended as a reusable OTP control

Primitive rules:
- no labels
- no descriptions
- no error message text
- no product-specific copy
- no business/domain-specific content
- own their own focus, hover, disabled, invalid/destructive, filled, and size states
- use token-bound geometry, typography, color, borders, radius, and shadows

## Structural Wrappers

Place reusable composition wrappers here:
- field
- compound control / grouped control shell, if present

Wrapper rules:
- field owns label, description/help text, error text, required/optional display if supported, and accessibility relationships
- compound control owns shared chrome/focus-within visual grouping for multiple controls
- wrappers do not own child values, options, validation timing, or product data

## Internal Parts

Place subcomponents here if they are not meant to be public package components:
- field label
- field description
- field card/control slot
- input counter button
- input counter group
- select chevron/checkmark atoms, if present

Internal-part rules:
- clearly name them as internal
- do not present them as standalone package components
- keep them token-bound

## Patterns / Playgrounds

Place product-like compositions here:
- field/text
- field/select
- field/number
- field/currency
- field/phone
- field/country
- card-input

Pattern rules:
- label them as Pattern, Playground, or Example
- do not name them as if they are exported package components
- show that they are composed from primitives and wrappers
- keep focus states as child-focus examples, not new product component APIs
- keep business/domain content in the example only

Specific cleanup guidance:

- If currency, phone, country, or card examples are currently componentized as standalone core components, rename/move them into Patterns / Playgrounds unless there is a deliberate decision to package them.
- If field/text, field/select, or field/number are matrices showing field + primitive combinations, classify them as examples, not primitives.
- If any primitive still contains label/description/error text, remove that from the primitive and represent it through field examples instead.
- If any example has focus states like focus-text or focus-select, keep those as documentation of child focus, not as public primitive variants.

Final report:

- What changed
- What was moved into Primitives
- What was moved into Structural Wrappers
- What was moved into Internal Parts
- What was moved into Patterns / Playgrounds
- Anything that still needs a design decision
```
