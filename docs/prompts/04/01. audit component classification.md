# Audit Component Classification

```text
You are working in the Design System v2.0 2026 Figma file.

Goal:
Audit the input components page against the design-system component model before making any changes.

Do not modify the file in this pass. Only inspect, classify, and report.

Context:
The code design system is built around:
- primitives
- structural wrappers
- stable behavior shells
- product patterns shown as examples/playgrounds, not exported package components

The component library manifest is:
- primitives own reusable behavior, visual identity, accessibility, states, and tokenized internal geometry
- structural wrappers own composition contracts such as label, description, error, and grouped layout
- product patterns are composed examples using primitives and wrappers

Examples of product patterns:
- currency amount field
- phone number field
- country selector
- card/payment input

These patterns should not become standalone package components unless they own stable cross-product behavior that would be error-prone to rebuild from smaller pieces.

Please audit the input components page and classify every relevant component or section as one of:

1. Primitive
A reusable component with one clear responsibility and durable behavior.

Examples:
- text input
- select input
- number input
- input OTP, if it owns OTP-specific keyboard/paste/autofill behavior
- button-like controls
- avatar-like display components

2. Structural wrapper
A reusable composition component that provides layout, state context, or accessibility wiring without owning business values.

Examples:
- field
- compound control / grouped control shell

3. Internal part
A subcomponent or atom used to build a primitive or wrapper, but not intended to be exported as its own public component.

Examples:
- input counter button
- input counter group
- field label
- field description
- field card/control slot

4. Pattern / playground
A product-like composition made from primitives and wrappers.

Examples:
- currency input
- phone input
- country input
- card input

Audit checklist:

- Identify every component/section on the input page.
- Classify each as Primitive, Structural wrapper, Internal part, or Pattern/playground.
- Flag anything that has mixed responsibility.
- Flag anything named like a core component but behaving like a product pattern.
- Flag anything with label, description, or error text inside a primitive control.
- Flag anything with product/domain content baked into a primitive.
- Flag anything that appears to duplicate a primitive instead of composing it.

Report format:

- Component or section name
- Classification
- PASS / NEEDS REVIEW / FAIL
- Reason
- Recommended action

Do not make changes until this audit is reviewed.
```
