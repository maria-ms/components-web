**Plan**

Run these prompts in order. Do not let the Figma Agent edit everything in one pass.

The overall goal is to clean up component sizing variables so the code package can consume generated tokens instead of hardcoded dimensions.

Right now Figma is correctly using variables for colors, typography, spacing, radius, borders, and shadows. However, several component heights, min-heights, approved widths, icon sizes, avatar sizes, status sizes, badge sizes, internal gaps, and padding values still appear to exist only as raw Figma layer dimensions. Raw dimensions do not export into the token package, so engineers cannot reference them as generated CSS variables.

The goal is not to duplicate values. The goal is to expose component intent in the exported variables.

Component variables should alias existing primitive or semantic variables where possible. For example, if 16px is `space/05`, then `component/input/icon/size` should alias `space/05`.

Use slash-separated Figma variable paths such as `component/input/icon/size`. Do not create source variables with hyphenated segments such as `component/input/icon-size`. Hyphenated CSS custom properties are generated later from slash-separated source paths.

Creating variables is not enough. The variables must be bound to the actual component geometry where Figma supports binding: icon frame sizes, avatar sizes, control heights, dropdown item heights, internal gaps, padding, and variant dimensions.

If Figma cannot bind a variable to a specific dimension, report that limitation instead of silently leaving the raw value in place.

**Geometry ownership policy**

Components own intrinsic geometry. Consumers own placement geometry.

Component-owned geometry includes control height, padding, internal gap, icon size, avatar size, OTP cell size, dropdown item min-height, focus ring, border radius, and other dimensions required for the component to look correct in isolation.

Consumer-owned geometry includes page/form width, max-width, grid/flex placement, columns, wrapping, and whether controls are arranged vertically or horizontally.

**Approved width token policy**

Width is not globally excluded. A design system should have approved width tokens where the system defines reusable sizes. The important distinction is where those width tokens are consumed.

Define and bind width variables when the width is a design-system-approved size, such as:
- canonical input/select sizes
- alert size variants
- dropdown menu surfaces
- attached control segments
- OTP cells
- reusable composition defaults

Do not turn page placement, artboard layout, or one-off example width into a component token.

Most width tokens should be consumed by parents, forms, field groups, Storybook examples, or application layouts. Primitive components should not force those widths internally. Primitive components should only own width when width is intrinsic to the component behavior or visual contract, such as OTP cell size, picker width matching a select trigger, a dropdown menu surface contract, or an attached control segment.
