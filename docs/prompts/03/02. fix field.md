# Fix Field

```text
You are working in the Design System v2.0 2026 Figma file.

Goal:
Fix the `field` component so it is production-ready for the design-system code pipeline.

Context:
The input primitives are now separated correctly. The field component should own label, description/help text, error text, required/optional display if supported, and shared field state. Primitive controls should not own labels/descriptions/errors.

Please make the following changes to `field`.

1. Clean up the required variant

If required state is part of the design system:
- Add `required=true`
- Keep `required=false`
- Ensure the required visual treatment is explicit and token-bound

If required state is not part of the design system:
- Remove `required` from the field component variant API

Do not leave a single-value variant like `required=false`.

2. Bind all field text to variables

Bind every text layer inside `field` to semantic/component variables.

Label:
- Typography should match `.input/label`
- Default color should use the same semantic variable as `.input/label`
- Disabled color should use a disabled semantic foreground variable

Description/help text:
- Typography should match `.input/description`
- Default color should use the same semantic variable as `.input/description`
- Disabled color should use a disabled semantic foreground variable

Error text:
- Typography should be intentionally defined
- Color should use a destructive/error semantic foreground variable

Do not leave raw hex colors or raw font values on field text layers where a variable exists.

3. Keep the control slot as a placeholder only

The dashed `control-slot` can stay as a Figma placeholder, but make sure it is clearly named/documented as a placeholder.

It should represent where consumers place real controls such as:
- text-input
- select-input
- number-input
- input-otp
- composed groups of controls

Do not turn `control-slot` into a new public primitive.

4. Verify composition examples

After fixing `field`, verify that these are still marked as patterns, not core primitives:

- country-input (Pattern)
- currency-input (Pattern)
- phone-input (Pattern)
- card-input (Pattern)

They may use field-like layout, but they should remain examples/recipes.

5. Final self-check

Before finishing, inspect `field` and confirm:

- No single-value required variant remains
- All field text colors are variable-bound
- All field text typography is variable-bound
- Destructive/error text uses destructive/error variables
- Disabled text uses disabled variables
- control-slot is clearly a placeholder
- The structure still maps cleanly to `ds-field`
```
